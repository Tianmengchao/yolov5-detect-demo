# YOLOv5 视觉检测学习路线

## 总体目标

从最简单的单张图片检测出发，逐步掌握模型部署、视频流处理、目标跟踪、摄像头接入、性能优化的完整链路。
最终形成工业级"检测 + 跟踪"Pipeline，每个阶段解决一个核心问题，通过 git commit 记录学习进展。

## 核心原则

1. **逐步引入变量** — 每一阶段只新增一个关键概念，出问题时能快速定位
2. **先跑通再优化** — 功能正确性优先，性能调优留到专门阶段
3. **最小依赖** — 能用标准库/轻量方案解决的，不引入重框架

## 阶段一：图片检测 ✅

**目标**：单张图片送入 RKNN 推理，输出标注结果图

**关键知识点**：
- RKNN 模型加载与初始化（`rknn_init` / `rknn_query`）
- 图像预处理：读图 → letterbox 缩放 → 像素格式转换
- NPU 推理：`rknn_inputs_set` → `rknn_run` → `rknn_outputs_get`
- 后处理：anchor 解码 → 置信度过滤 → NMS 去重
- 结果可视化：画框 + 写出 PNG

**验证标准**：
- `./yolov5_detect_demo model/yolov5.rknn model/bus.jpg` 生成 `out.png`
- 检测框位置合理，类别标签正确

---

## 阶段二：视频文件逐帧检测 ← 当前

**目标**：读取本地视频文件，逐帧送入同一推理流程，输出结果视频

**为什么从视频文件开始而非摄像头**：
- 输入可控、可复现，排查问题时无需反复触发场景
- 避免同时面对：摄像头驱动 / V4L2 / 分辨率协商 / 像素格式 / 帧率 / 缓冲区管理

**关键知识点**：
- OpenCV `VideoCapture` 打开视频文件，逐帧读取
- BGR → RGB 颜色空间转换（OpenCV 默认 BGR，RKNN 需要 RGB）
- `cv::Mat` 与 `image_buffer_t` 之间的数据桥接
- `VideoWriter` 将标注后的帧写入结果视频
- 推理循环的基本结构：`while (cap.read(frame)) { ... }`

**实现思路**：
```
打开视频文件
初始化 RKNN 模型（仅一次）
循环 {
    读取一帧 (cv::Mat, BGR)
    BGR → RGB 转换
    填充 image_buffer_t 结构
    调用 inference_yolov5_model()
    在帧上绘制检测结果
    写入输出视频 / 显示窗口
}
释放资源
```

**新增依赖**：OpenCV（视频编解码、帧操作）

**验证标准**：
- 输入一段包含行人/车辆的测试视频
- 输出视频中每帧都有正确的检测框
- 帧率信息打印（了解当前单帧推理耗时）

## 阶段三：RTSP 流实时检测

**目标**：拉取网络 RTSP 视频流，实时推理并显示/输出

**为什么用 RTSP 替代 USB 摄像头作为实时流入门**：
- 当前开发环境暂无 USB 摄像头，但有可用的 RTSP 流源
- RTSP 同样是"无限实时流"，能练习与阶段二相同的关键差异点（丢帧、延迟、超时处理）
- 网络流更贴近工业部署场景（IP 摄像头、NVR 接入）

**关键知识点**：
- OpenCV `VideoCapture("rtsp://...")` 拉流（底层走 FFmpeg）
- 网络流 vs 本地文件的关键差异：
  - 无法 seek，只能顺序消费
  - 网络抖动导致帧到达不均匀
  - 读取阻塞 / 超时处理
- 丢帧策略：当推理速度跟不上流速时，丢弃旧帧只处理最新帧
- TCP vs UDP 传输模式选择（`rtsp_transport` 参数）
- 解码延迟：FFmpeg 内部缓冲区管理

**实现思路**：
```
打开 RTSP 流 (cv::VideoCapture + RTSP URL)
设置低延迟参数 (降低内部缓冲)
初始化 RKNN 模型（仅一次）
循环 {
    grab 最新帧（丢弃积压帧）
    BGR → RGB + 填充 image_buffer_t
    推理 + 后处理
    绘制结果
    显示 / 写入文件
    打印帧率和延迟
}
```

**验证标准**：
- 能稳定拉取 RTSP 流并逐帧检测
- 处理网络中断后自动重连
- 帧率信息打印，了解端到端延迟
- 无内存泄漏（可长时间运行 >10 分钟）

## 阶段三-B：USB 摄像头实时检测（待硬件就绪）

> 当前暂跳过，等 USB 摄像头到位后再回来补充。
> 核心逻辑与阶段三高度复用，主要差异在输入源初始化。

**目标**：接入 USB 摄像头，实时推理并显示

**关键知识点**：
- V4L2 基础：`/dev/video*` 设备枚举
- 摄像头参数协商：分辨率、帧率、像素格式（YUYV / MJPEG）
- OpenCV `VideoCapture(0)` 快速接入（封装了 V4L2）
- 实时显示：帧率控制、延迟感知

**与 RTSP 流的额外差异**：
- 本地设备，无网络延迟，但有 V4L2 驱动层面的复杂性
- 可能遇到的坑：摄像头默认 YUYV 格式需额外转换
- 帧率由硬件能力决定，不受网络带宽限制

**验证标准**：
- 插入 USB 摄像头后能正常出图
- 实时检测帧率 ≥ 5 FPS（在未优化条件下的基线）
- 无内存泄漏（可长时间运行）

## 阶段四：目标跟踪 — ByteTrack

**目标**：在逐帧检测基础上，为每个目标分配持久 ID，实现跨帧跟踪

**为什么在此阶段引入跟踪**：
- 工业场景中，纯检测只能回答"这帧有什么"，跟踪才能回答"这个目标是谁、从哪来、到哪去"
- 计数（人流/车流）、轨迹分析、越界告警、行为判定全部依赖跟踪
- ByteTrack 是纯 CPU 算法（卡尔曼滤波 + 匈牙利匹配），不需要额外 NPU 资源，适合嵌入式

**关键知识点**：
- 跟踪 vs 检测的本质区别：检测是帧内问题，跟踪是帧间关联问题
- 卡尔曼滤波（Kalman Filter）：预测目标下一帧位置
- IoU 匹配 + 匈牙利算法（Hungarian Algorithm）：检测框 ↔ 轨迹的最优分配
- ByteTrack 核心创新：对低置信度检测框做二次匹配，减少 ID Switch
- Track 生命周期管理：新建（Tentative）→ 确认（Confirmed）→ 丢失（Lost）→ 删除

**实现思路**：
```
每帧 {
    运行 YOLOv5 检测 → detections[]
    卡尔曼滤波预测所有已有 track 的当前位置
    第一次匹配：高置信度 detections ↔ 现有 tracks（IoU）
    第二次匹配：低置信度 detections ↔ 未匹配 tracks（ByteTrack 特色）
    未匹配的 detection → 新建 track
    长时间未匹配的 track → 标记删除
    输出：每个目标的 (bbox, class, track_id)
}
```

**新增依赖**：Eigen（矩阵运算，卡尔曼滤波用）或手写轻量版

**验证标准**：
- 同一目标跨帧保持相同 ID（在视频中用不同颜色区分 track）
- 目标短暂遮挡后 ID 不丢失
- 能统计视频中出现的目标总数（去重计数）
- ID Switch 率低（同一目标不频繁换 ID）

**进阶方向**（本阶段不做，后续可选）：
- DeepSORT：引入 ReID 外观特征模型，应对长时遮挡和密集场景
- BoT-SORT：结合摄像头运动补偿（CMC），适合移动平台
- OC-SORT：虚拟轨迹插值，减少遮挡场景的 ID Switch

## 阶段五：性能基线与瓶颈分析

**目标**：测量完整 pipeline（检测 + 跟踪）各环节耗时，找出瓶颈

**关键知识点**：
- 计时埋点：预处理 / 推理 / 后处理 / 跟踪 / 绘制 各多少 ms
- `rknn_query(RKNN_QUERY_PERF_DETAIL)` 获取 NPU 层级耗时
- 帧率瓶颈在 CPU 预处理还是 NPU 推理？跟踪耗时占比如何？
- 内存带宽：大图反复 malloc/free vs 预分配复用

**验证标准**：
- 能输出清晰的各环节耗时报告
- 明确知道"当前帧率受限于哪个环节"

## 阶段六：预处理优化 — RGA 硬件加速

**目标**：用 RGA 替代 CPU 进行图像缩放和格式转换

**关键知识点**：
- RGA（Rockchip Graphics Acceleration）：硬件 2D 加速器
- `imresize` / `imcvtcolor`：替代 OpenCV resize + cvtColor
- DMA Buffer：零拷贝传递图像数据
- RGA 对齐要求：宽度 4/16 对齐

**验证标准**：
- 预处理耗时显著下降（对比阶段五的基线）
- 输出结果与 CPU 预处理一致

---

## 阶段七：异步 Pipeline — 多线程流水线

**目标**：将采集、预处理、推理、跟踪、后处理解耦为多级流水线

**关键知识点**：
- 生产者-消费者模型
- 环形缓冲区（Ring Buffer）在各级间传递帧
- `std::thread` + `std::mutex` + `std::condition_variable`
- 推理与采集并行：NPU 推理第 N 帧时，CPU 同时在预处理第 N+1 帧
- 跟踪必须串行（依赖帧序），但可以和下一帧的预处理并行

**流水线结构**：
```
[采集线程] → [预处理线程] → [推理线程] → [跟踪+后处理线程] → [显示/编码线程]
     ↓              ↓              ↓              ↓                  ↓
  Ring Buffer   Ring Buffer   Ring Buffer   Ring Buffer           输出
```

**验证标准**：
- 端到端帧率相比阶段六有提升
- 各线程负载均衡（无某一级长期阻塞）
- 跟踪 ID 连续性不受多线程影响

---

## 阶段八：MIPI 摄像头与硬件编解码

**目标**：接入 MIPI CSI 摄像头，使用 MPP 硬件编解码

**关键知识点**：
- MIPI CSI 接口 vs USB：带宽更高、延迟更低、配置更复杂
- ISP（Image Signal Processor）：RAW → YUV 的硬件处理
- Rockchip MPP（Media Process Platform）：硬件编解码 H.264/H.265
- V4L2 subdev 架构：sensor → ISP → 输出多路流

**验证标准**：
- MIPI 摄像头出图正常
- 使用 MPP 编码器将检测结果编码为 H.264 视频流

---

## 阶段九：系统集成与工程化

**目标**：形成可部署的完整"检测 + 跟踪"应用

**可选方向**（根据实际需求选择）：
- RTSP 推流：结合 live555 / GStreamer 对外提供视频流服务
- DRM 直显：绕过 X11/Wayland，直接输出到屏幕（最低延迟）
- 多模型切换：支持 YOLOv5 / YOLOv8 / 自训练模型热切换
- 配置化：命令行参数控制分辨率、置信度阈值、输入源
- 跟踪增强：接入 ReID 模型（DeepSORT），应对密集遮挡场景
- 业务逻辑：越线计数、区域入侵告警、轨迹热力图
